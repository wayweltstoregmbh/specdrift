# Analyse & v0.2-Plan: Wo stehen wir, was bauen wir — und was bewusst nicht

Datum: 2026-06-12
Leitfrage: Welche Features erweitern real die Kompetenz von Agenten (und
ihrer Nutzer) so, dass ein Abo Sinn ergibt — ohne Fokusverlust, ohne
Overengineering, immer testgetrieben und kleinschrittig?

---

## Teil 1: Standortbestimmung (nüchtern)

### Was existiert und bewiesen ist

| Baustein | Status | Beweis |
| --- | --- | --- |
| OSS-Kern M0–M5 (CLI, MCP, Action) | fertig | 79 Tests, 3-OS-CI grün, CI-Wirksamkeit bewiesen |
| Drift-Fang real | bewiesen | Echte WP-Spec, eingebauter Drift → Gate nennt Claim+Zeile |
| Bedienbarkeit | bewiesen | Blind-Test durch kontextfreien Agenten nur mit README; 7 Funde behoben |
| Benchmark-Gate | aktiv | ≥90 % Recall/Precision als Release-Bedingung |
| Cloud-Kern (M6) | läuft lokal | Hash-Kette, Keys, Isolation — Live-Smoke + Portal-E2E |
| Portal (M7-Basis) | läuft lokal | Landing/Pricing/Docs/Konto, Registrierung→Key→Upload E2E |

### Die drei ehrlichen Schwächen (aus den Praktiker-Fragen)

1. **Iterations-Lücke:** Spec-Erweiterung invalidiert ALLE Verdicts
   (positionsbasierte Claim-IDs + globaler Hash). Iteratives Arbeiten — der
   Normalfall — ist damit unnötig teuer.
2. **Chat-Lücke:** Anforderungen, die nur im Prompt entstehen, erreichen die
   Spec nie. Nur das Symptom (Änderung ohne Spec-Deckung) wird erkannt.
3. **Wahrheits-Lücke:** SpecDrift erzwingt Rechenschaft, nicht Wahrheit. Ein
   unehrliches `covered` mit existierender, aber unpassender Evidenz wird
   nicht maschinell erkannt.

### Die Kompetenz-These (der Abo-Grund)

Ein Agent mit SpecDrift kann vier Dinge, die er ohne nicht kann:

- **Nicht vergessen:** Das Claim-Hauptbuch überlebt Kontext-Kompaktierung
  und Session-Ende — Anforderungen verschwinden nicht mehr im Kontextfenster.
- **Wiederaufsetzen:** Eine frische Session kann exakt sehen, was offen,
  erledigt, gewaivert ist — statt die Historie zu rekonstruieren.
- **Beweisen:** Jede „Fertig"-Behauptung zeigt auf eine prüfbare Stelle.
- **Gegated werden:** Der Merge wartet, bis das Hauptbuch ausgeglichen ist.

**Fokus-Regel für jede Feature-Entscheidung:** SpecDrift ist *das Hauptbuch
zwischen Spec und Merge*. Ein Feature kommt nur rein, wenn es (a) die
Vertrauenswürdigkeit dieses Hauptbuchs erhöht oder (b) seine Benutzbarkeit
für Agenten im Alltag erhöht. Alles andere ist Scope-Drift — ironischerweise
genau das, wogegen wir das Produkt bauen.

## Teil 2: Priorisierung — was jetzt, in dieser Reihenfolge

Begründungslogik: Die Iterations-Lücke killt Alltagsnutzung → zuerst. Die
Chat-Lücke ist die größte Kompetenz-Erweiterung für Agenten → direkt danach
(baut auf 1 auf). Wiederaufsetzen ist billig und hoher Agentennutzen → dritter.
Wahrheits-Heuristik macht das Hauptbuch vertrauenswürdiger → vierter. Cloud-
Vervollständigung verbindet OSS mit dem Abo → fünfter. Bezahlung kommt erst,
wenn es Nutzer gibt, die zahlen wollen.

### S1 — Stabile Claim-Identität (≈ 1–2 Tage) → schließt Lücke 1a

Claim-IDs werden aus dem Inhalt abgeleitet (Hash über normalisierten
Anforderungstext + Sektionspfad) statt aus der Position. Re-Extraktion nach
Spec-Änderung erhält die IDs unveränderter Claims.

- Positiv: Claim behält ID, wenn oberhalb eine neue Anforderung eingefügt
  wird; identischer Text in zwei Sektionen → zwei verschiedene stabile IDs;
  Determinismus bleibt.
- Negativ: geänderter Anforderungstext → neue ID (alte erscheint als
  entfernt); Duplikat-Statement in derselben Sektion → deterministischer
  Suffix, kein Crash; alte positionsbasierte Stores werden als
  Altformat erkannt (Migrationshinweis statt stiller Fehlinterpretation).

### S2 — `extract --update` mit Verdict-Erhalt + Delta (≈ 2 Tage) → schließt Lücke 1b

Re-Extraktion in einen bestehenden Store: unveränderte Claims behalten ihre
Verdicts, neue Claims erscheinen als `unverdicted`, entfernte werden mit
ihren Verdicts archiviert statt gelöscht. Ausgabe ist ein **Delta-Report**
(added/removed/unchanged/changed).

- Positiv: Spec um 2 Anforderungen erweitert → 38 Verdicts überleben, Report
  zeigt exakt 2 offene; Delta-Report deterministisch.
- Negativ: geänderter Claim-Text → genau dieser Verdict invalidiert, Rest
  bleibt; `--update` ohne bestehenden Store → klarer Fehler; Gate bleibt rot,
  solange neue Claims offen sind (kein False-Green durch Update).

### S3 — `spec add` + MCP-Tool `add_requirement` (≈ 1–2 Tage) → schließt Lücke 2

Der formelle Weg vom Prompt ins Hauptbuch: Anforderungstext + Zielsektion →
wird an die Spec-Datei angehängt, Update-Extraktion läuft, Delta kommt
zurück. Der Agent bekommt damit den Reflex „neue Nutzeranforderung →
sofort protokollieren", mit Quellenanker statt Kontext-Hoffnung.

- Positiv: Mid-Session-Anforderung → Spec-Datei erweitert, neuer Claim mit
  korrektem Anker, alte Verdicts intakt; MCP-Antwort enthält Delta.
- Negativ: leerer Text / nicht existente Sektion → abgelehnt; Non-Goal als
  Zielsektion → abgelehnt (Grenzen sind keine Anforderungen); Schreibfehler
  auf read-only Spec → sauberer Fehler ohne halben Zustand.

### S4 — MCP-Tool `get_open_work` (≈ 0,5–1 Tag) → Kompetenz „Wiederaufsetzen"

Eine Abfrage liefert den Arbeitsstand: offene Claims (mit Text+Zeile),
Drift-Posten, Waiver, letzte Evidenzen. Eine frische Agent-Session setzt
exakt dort auf, wo die letzte endete — Daten existieren alle schon, das ist
fast nur Projektion.

- Positiv: nach Teil-Verdicts liefert das Tool exakt die offene Restliste;
  leerer Stand → „alles abgerechnet" statt Fehler.
- Negativ: stale Spec → Tool meldet Staleness als ersten Schritt statt
  veralteter Liste; kein Store → Hinweis auf extract.

### S5 — Verdict-Plausibilitätsprüfung (≈ 1–2 Tage) → verkleinert Lücke 3 ehrlich

`verify-verdicts`: deterministische Heuristik prüft jedes `covered` —
existiert die Evidenz-Datei/Zeile, hat sie Token-Überlappung mit dem Claim?
Ergebnis ist eine **Verdachtsliste** („evidence_suspicious"), kein Urteil.
Im Gate optional als Warnung, per Flag blockierend. Bewusst keine LLM-Calls;
die tiefe Prüfung bleibt dem Reviewer/zweiten Agenten überlassen — aber die
Verdachtsliste sagt ihm, WO er hinschauen soll.

- Positiv: covered mit themenfremder Evidenz-Zeile → suspicious; echtes
  covered → unauffällig (Benchmark-Erweiterung mit geseedeten Fake-Verdicts,
  Ziel ≥80 % Verdachts-Recall bei ≤10 % False-Positives).
- Negativ: gelöschte Evidenz-Datei nach Verdict → suspicious mit Grund;
  Heuristik crasht nie auf Binär-/Riesendateien.

### S6 — `specdrift push` + M6-Testbatterie (≈ 2–3 Tage) → verbindet OSS und Abo

CLI-Kommando lädt Reports in den Cloud-Workspace (Key aus Umgebungsvariable,
nie aus Dateien). Dazu die noch offene M6-Pflicht aus TESTPLAN.md:
Manipulations-Erkennung (DB-Tampering → Kettenbruch), Widerruf < 60 s,
Rate-Limit 429, Oversize 413, Isolation — als automatisierte Suite gegen die
lokale Instanz. Chain-Head-Anchoring kommt erst vor der Beta (braucht erst
echte Daten, die es zu schützen lohnt).

### Danach: OSS-Launch (npm publish, v0.2-Release, Show HN) — DANN erst Beta.

## Teil 3: Was wir bewusst NICHT bauen (Anti-Overengineering-Liste)

| Versuchung | Warum nicht jetzt |
| --- | --- |
| WooCommerce/Stripe-Checkout | Kein einziger Beta-Nutzer existiert. Die ersten 5 zahlenden Teams bekommen manuelle Rechnungen — das prüft Zahlungsbereitschaft ohne eine Zeile Checkout-Code. Checkout erst nach Markt-Gate M8. |
| Chat-Transkript-Mining (Anforderungen automatisch aus Prompts extrahieren) | `add_requirement` löst 90 % des Problems mit 5 % des Aufwands und hält den Menschen in der Schleife. |
| LLM-basierte Verdict-Prüfung | Bricht das Null-Kosten/Determinismus-Versprechen. Heuristik + Verdachtsliste zuerst; LLM-Judge ist v0.3-Kandidat, falls Nutzer es fordern. |
| Dashboard-Ausbau, Charts, Trends | Der Report-Viewer existiert; Visualisierung verkauft kein Vertrauen. |
| Weitere Spec-Formate (reST, AsciiDoc, Notion) | Erst wenn echte Nutzer daran scheitern. Issues abwarten. |
| Reverse-Drift voll (`reverse --diff`) | Uncovered-Change-Warnung im Gate deckt den akuten Fall; Vollausbau nach Beta-Feedback. |
| Compliance-PDF-Exporte | Kein regulierter Kunde vorhanden. JSON/CSV-Export reicht der Beta. |
| Org-Umzug, Branding, Logo | Kostet Fokus, bringt keinen Nutzer. Nach dem Launch. |

## Teil 4: Arbeitsregeln (unverändert verbindlich)

1. Jeder Schritt S1–S6 wird einzeln umgesetzt, mit gepaarten
   Positiv-/Negativ-Tests (IDs `V2-S<x>-P/N<nn>`, Anhang an TESTPLAN.md),
   grüner kumulierter Regression und eigenem Commit — kein Schritt beginnt,
   bevor der vorherige abgenommen ist.
2. Das Benchmark-Gate (≥90/90) bleibt Release-Bedingung; S5 erweitert das
   Benchmark um geseedete Fake-Verdicts.
3. Kein neues npm-Paket, keine neue Dependency ohne explizite Entscheidung.
4. Jede Abweichung von diesem Plan wird hier dokumentiert, bevor sie
   implementiert wird — wir führen unser eigenes Hauptbuch.

## Teil 5: Erfolgsmessung

- **v0.2 ist fertig, wenn:** das Iterations-Szenario (Spec dreimal
  erweitern, dazwischen implementieren) ohne Verdict-Massenverlust
  durchläuft, ein Agent per MCP eine Chat-Anforderung formell protokollieren
  kann, eine frische Session per `get_open_work` korrekt wiederaufsetzt, und
  die Fake-Verdict-Erkennung ihre Benchmark-Ziele hält.
- **Aufwand gesamt:** ≈ 8–11 Entwicklungstage bis v0.2 + Launch-Reife.
- **Danach gilt wieder das Markt-Gate aus der Roadmap:** Installs/Feedback
  entscheiden über Beta und Bezahlausbau — nicht unser Bauchgefühl.
